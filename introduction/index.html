Blazor components for application UI

Build shadcn-style interfaces in Razor.

Quark Suite gives Blazor apps a practical component layer: styled components when you want defaults, primitives when you need control, and typed Tailwind helpers when raw class strings are not enough.

What Quark is

Quark provides Razor components for common application UI: buttons, dialogs, forms, layout primitives, tables, navigation, typography, and data-display components. It is meant for product interfaces where the markup needs to stay understandable and the API should feel native to Blazor.

The components render ordinary HTML elements where possible and expose typed parameters for layout, spacing, color, typography, accessibility attributes, and component-specific behavior. You can stay in the component API for common work and drop down to plain HTML, classes, styles, and attributes whenever the application needs it.

Why use it

Accessible shape

Components keep semantic elements, ARIA hooks, and keyboard behavior close to the surface instead of burying them behind opaque wrappers.

C# ergonomics

Variants, sizes, layout, and many utilities are available as typed parameters, so refactors get compiler help and editor completion.

Design-system friendly

Theme tokens and utility classes are first-class, so teams can keep shadcn-compatible defaults or replace them with their own system.

Build-time support

Generators collect Tailwind utilities, icon usage, and optional theme CSS so the app owns the output it ships.

Relationship to shadcn/ui

shadcn/ui is the reference point for many component APIs, variants, slots, and visual contracts. Quark ports that style of component composition to Blazor, while using Razor parameters and .NET types instead of React props.

When a component maps to a shadcn/ui pattern, Quark keeps the naming and behavior recognizable. When Blazor needs a different shape, Quark favors Razor-native APIs over a forced React copy.

How the pieces fit

Components

Use components when you want finished UI with consistent defaults: buttons, dialogs, fields, tables, navigation, feedback, and data display. They are the fastest path for screens that should already look like part of the system.

Primitives

Use primitives when the behavior matters but the visual treatment belongs to your app. They keep composition flexible for custom surfaces, internal design systems, and edge cases that should not fight a prebuilt component.

Relationship to Tailwind

Tailwind CSS is the styling layer. Quark builder parameters emit Tailwind utility classes, and the demo/app build uses Quark generators to collect the utilities used by Razor files and produce the app stylesheet.

You can use typed builders for common utilities, raw Class values for exact Tailwind selectors, and theme tokens for shadcn-compatible colors. This keeps simple styling typed without blocking advanced Tailwind usage.

Stack and theming

Blazor and Razor

Components are authored for .NET apps and favor Razor-native APIs over React-shaped copies.

Tailwind utilities

Styling remains inspectable because utilities are generated into the app stylesheet.

CSS variables

Theme values flow through shadcn-style tokens for light, dark, sidebar, series, and surface colors.

OKLCH palettes

Generated Tailwind colors use modern color tokens while still compiling down to ordinary CSS.

What Quark gives you

  • Blazor components modeled after shadcn/ui patterns where those patterns make sense.
  • Typed Tailwind builders for layout, spacing, typography, color, borders, sizing, effects, and responsive modifiers.
  • Semantic primitives such as Div, Section, Paragraph, Anchor, Grid, and Stack.
  • Escape hatches through Class, Style, ARIA attributes, data attributes, and unmatched Razor attributes.
  • Build-time generation for Tailwind output, icon usage, and optional shadcn-compatible theme CSS.
